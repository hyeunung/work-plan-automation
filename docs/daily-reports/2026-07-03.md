## 📢 07월 03일 (금요일) 업무 요약 브리핑

> 각 팀원들의 금일 업무 목적 및 진행 의미 요약입니다.

### 👤 김윤회 님
* **부품 입고**
  - *상세*: 입고 확인(7/2)
* **시스템 리팩토링 - 클럭 구조 안정화 착수**
  - *목적*: 캡처 제어 신호와 상태 readback이 서로 다른 클럭 영역을 건너며 생길 수 있는 불안정성을 구조적으로 제거
  - *진행 의미*: 우연히 통과하는 조합에 의존하지 않고, 향후 상태 표시 기능과 캡처 제어가 안정적으로 동작할 수 있는 하드웨어 기반을 만드는 단계
* **시스템 리팩토링 - 비트스트림 검증 및 빌드 흐름 안정화**
  - *목적*: 같은 소스라도 빌드 산출물과 부팅 경로에 따라 동작이 달라지는 문제를 줄이고, 재현 가능한 검증 흐름 확보
  - *진행 의미*: 장애가 재발했을 때 GUI, 펌웨어, 비트스트림, 부팅 이미지 중 어느 단계 문제인지 더 빠르게 좁힐 수 있는 기준 확보
* **시스템 리팩토링 - 보드별 캡처 GUI 개선**
  - *목적*: 여러 보드 운용 시 전역 설정과 보드별 테스트 설정이 섞이지 않게 하고, 캡처 실패 원인을 화면에서 더 쉽게 확인
  - *진행 의미*: 현장 테스트 중 캡처가 멈췄는지, 생성기가 아직 동작 중인지, 보드별 설정이 무엇인지 구분하기 쉬운 운용 화면으로 개선
* **시스템 리팩토링 - 대용량 캡처 구조 설계**
  - *목적*: 최대 20개 보드, 보드당 6채널, 채널당 100만 pulse 수준의 캡처를 안정적으로 처리하기 위한 방향 정리
  - *진행 의미*: 현재 단일보드 검증 단계에서 9보드, 20보드 운용으로 확장할 때 필요한 HW/FW/GUI 변경 범위를 사전에 정리
* **시스템 리팩토링 - 캡처 종료 장애 원인 분석**
  - *목적*: GUI에서 캡처 정지 후 다운로드가 끝나지 않는 문제와 부팅 시 내부 loopback 자체 테스트 실패 원인을 분리 확인
  - *진행 의미*: 캡처 timeout을 단순 GUI 문제가 아니라 부팅·비트스트림·클럭 구조까지 포함한 하드웨어 검증 문제로 정리하고 재발 확인 절차를 확보
